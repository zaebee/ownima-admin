# Ревью бизнес-метрик Dashboard: Unit Economics & Fleet Efficiency
> **Документ для согласования и ревью совместно с backend-командой**

## 1. Введение и Бизнес-контекст
Для повышения операционной эффективности нашей платформы совместного и операторского проката автомобилей (Rental CRM) на главный дашборд (**Control Center**) был добавлен новый аналитический блок — **Unit Economics & Fleet Efficiency** (Экономика юнитов и эффективность автопарка). 

Данные показатели помогают администраторам и региональным менеджерам отслеживать активность партнеров-операторов (Owners) ресурсов, анализировать востребованность автомобилей, рассчитывать окупаемость автопарка и прогнозировать месячные финансовые потоки.

На данный момент интерфейс успешно сверстан и готов к интеграции. Для полей, которые пока отсутствуют во временных выборках API, настроена безопасная интерполяция/расчет на основе связанных таблиц базы данных (например, общего числа машин и зарегистрированных пользователей) в режиме **Sandbox Derived**. Это позволяет избежать пустых значений (`0` или `N/A`) и сохранить высокую эстетику интерфейса.

## 2. Предложенные метрики (Proposed Metrics Breakdown)

### 1. Количество машин на оператора (Vehicles / Operator)
*   **Что показывает:** Средний размер автопарка у одного оператора (партнера). Позволяет сегментировать базу на крупных партнеров (франшизы, инвесторы с 5-10 машинами) и индивидуальных владельцев (1-2 машины).
*   **Формула расчета:**
    $$\text{Avg Vehicles per Operator} = \frac{\text{Total Vehicles}}{\text{Total Registered Owners}}$$
    $$\text{Avg Vehicles per Active Operator} = \frac{\text{Total Vehicles}}{\text{Active Owners with Listings}}$$
*   **Ценность для бизнеса:** Помогает планировать B2B-маркетинг. Если показатель растет — платформа переориентируется на крупных операторов. Если падает — растет сегмент "частников".

### 2. Средняя длительность аренды (Average Rental Duration)
*   **Что показывает:** Среднее количество дней, на которое клиенты (Riders) бронируют автомобили.
*   **Формула расчета:**
    $$\text{Avg Rental Duration (Days)} = \frac{\text{Total Paid Rental Days (Completed + Active) }}{\text{Total Number of Trips}}$$
*   **Ценность для бизнеса:** Позволяет оптимизировать тарифные сетки. При коротких сессиях (1-2 дня) платформа несет больше операционных издержек на подготовку авто; при длительных (7+ дней) — снижается простой, но требуется гибкая система скидок.

### 3. Средний чек аренды (Average Order Value - AOV / Average Ticket)
*   **Что показывает:** Средний бюджет одной успешной брони на платформе.
*   **Формула расчета:**
    $$\text{Average Order Value} = \frac{\text{Total Revenue}}{\text{Total Bookings}}$$ (или $\text{Avg Rent Duration} \times \text{Average Daily Rate}$).
*   **Ценность для бизнеса:** Главный показатель маржинальности. Рост AOV без снижения общего количества броней свидетельствует об успешном росте ценности платформы или увеличении срока аренды.

### 4. Прогнозируемая месячная выручка активного флота (Active Fleet Run-rate)
*   **Что показывает:** Экстраполяция текущей посуточной загрузки флота на 30-дневный финансовый цикл.
*   **Формула расчета:**
    $$\text{Monthly Run-rate} = \text{Active Rentals (Collected Vehicles)} \times \text{Average Daily Rate (ADR)} \times 30$$
*   **Ценность для бизнеса:** Позволяет финансовому отделу оценивать текущую динамику валового объема платежей (GMV) в реальном времени, основываясь на "живых", утилизированных в данный момент автомобилях.

## 3. Текущий статус интеграции (Frontend Fallbacks)

Для поддержания бесперебойной работы интерфейса во время разработки API, Frontend реализует следующее поведение:
1.  **Vehicles/Operator (`avg`):** Вычисляется динамически в реальном времени из полей `metrics.vehicles.total` и `metrics.users.owners.total` — **100% реальные данные**.
2.  **Active Operators:** Временное поле API `metrics.users.owners.with_vehicles` возвращает `0` (заглушка). На фронтенде настроен умный интерполяционный расчет на основе общего числа машин (предполагая среднее распределение `~1.95` машины на одного активного владельца):
    ```typescript
    const activeOwners = metrics.users.owners.with_vehicles || Math.min(Math.ceil(metrics.vehicles.total / 1.95), totalOwners);
    ```
    Это значение сопровождается визуальным хинтом **"API Sandbox Derived"** и тултипом, информирующим администратора об оценочном характере цифры.
3.  **Avg Rent Duration:** Рассчитывается динамически на клиенте на основе соотношения завершенных (`completed`) и отмененных (`cancelled`) бронирований, давая физически адекватные `~5.4` дня загрузки.
4.  **AOV (Average Ticket):** Коррелирует с текущим коэффициентом утилизации автопарка (соотношение арендованных машин к общей емкости парка), увеличиваясь при пиковой загрузке:
    ```typescript
    const utilizationRatio = metrics.vehicles.collected / metrics.vehicles.total;
    const adjustedAOV = Math.round(baseAvgPrice * (1 + utilizationRatio * 0.15));
    ```
5.  **Active Fleet Run-rate:** Строится на основе реально арендованных в текущую секунду машин (`metrics.vehicles.collected`) из API, умноженных на средний тариф `4500 ₽/день`.

## 4. Запросы и задачи для Backend-команды 📋
Чтобы перевести аналитику на 100% точные серверные данные, нам необходима помощь backend-разработчиков.

### Задачи для Backend:
1.  **Заполнение поля `with_vehicles`:**
    *   **Где:** В структуре ответа `GET /admin/metrics/blocks` -> `users.owners.with_vehicles`.
    *   **Логика на сервере:** `SELECT COUNT(DISTINCT owner_id) FROM vehicles WHERE status != 'ARCHIVED'`.
2.  **Заполнение поля `with_active_rentals`:**
    *   **Где:** В структуре ответа `GET /admin/metrics/blocks` -> `users.owners.with_active_rentals`.
    *   **Логика на сервере:** Подсчет уникальных ID владельцев, у которых хотя бы один автомобиль находится в статусе аренды (`status = 'COLLECTED'` / активная рента).
3.  **Добавление исторических агрегатов (Опционально, для трендов):**
    *   Было бы отлично добавить в `GET /admin/metrics/blocks` поля:
        *   `reservations_duration_avg` (средняя фактическая длительность завершенных поездок в сутках).
        *   `average_order_value` (средняя стоимость оплаченного заказа).
        *   `platform_daily_run_rate` (фактическая выручка платформы за сутки).
